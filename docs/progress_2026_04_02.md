## 2. Diagnostic Analysis (Today)

### 2.1 LCB Landscape Inspection (1D Slice)

- TabPFN LCB and GP LCB are on the **same scale** — TabPFN is not hallucinating wrong magnitudes
- TabPFN LCB is **wiggly / non-smooth** compared to GP's smooth kernel-defined landscape
- Root cause: TabPFN's ensemble (4 estimators, transformer-based) is not kernel-smoothed;
  each estimator interpolates differently in sparse 50D regions → large-amplitude, high-frequency
  disagreement between estimators → wiggly ensemble mean

### 2.2 Distill MLP Collapse — Root Cause

Both observations (wiggly LCB + distill collapse) share the same root cause:

> **TabPFN's predictions in sparse high-d regimes are not reliable enough to optimise against.**

Distill failure chain:
```
n_distill=200 points in 50D → near-zero coverage
→ MLP fits smooth approximation of wiggly landscape (loses all structure)
→ L-BFGS-B extrapolates far from training support
→ proposed x is in a region TabPFN never saw → actual f(x) is bad
```

Suspected signature: check whether distill-proposed points cluster near the **boundary** of [0,1]^50
— this is the extrapolation fingerprint.

### 2.3 CMA-ES as Pending Diagnostic

CMA-ES is robust to non-smooth landscapes (evaluates fitness, no gradients).
Two possible outcomes when results arrive:

| Outcome | Interpretation |
|---|---|
| CMA-ES recovers TFM to MTBO/BO level | Acquisition optimizer was the bottleneck; surrogate is fine in 50D |
| CMA-ES still collapses | TabPFN surrogate itself is unreliable at 50D; no optimizer can fix a bad landscape |

**Updated prior:** given the wiggly landscape, CMA-ES may partially recover but not fully close
the gap — a wiggly landscape gives CMA-ES inconsistent fitness signals, preventing covariance
matrix adaptation from converging reliably.

## 3. Methodological Reflection

### 3.1 Why TFM-as-Surrogate Fails

TabPFN was not designed to produce a smooth, calibrated uncertainty landscape for continuous-space
optimisation. The failure is **structural**, not a tuning issue:

- Non-smooth predictions → candidate-pool optimisation finds local wiggles, not true optima
- Not differentiable w.r.t. input x → gradient-based acquisition (L-BFGS-B) cannot be used directly
- Sparse 50D data → ensemble disagreement is large → wiggles dominate the useful signal

### 3.2 Where TabPFN is Actually Strong

- **Relative ranking** of candidates (pre-trained on millions of tabular datasets)
- **Cross-task prediction**: assessing whether task t1's data predicts task t2's values well
- **Few-shot function characterisation**: capturing the "type" of a function from a handful of observations
- None of these require smoothness or differentiability — they are ranking/classification capabilities

## 4. New Research Proposals (Written Today)

Full proposal at: `docs/research_proposal_tfm_bo.md`

### Method 1: TFM-Screen (TabPFN as Acquisition Pre-Filter)

Use TabPFN to rank 10K LHS candidates → select top-200 → run standard GP `optimize_acqf` on
those 200. TabPFN does ranking (robust); GP does smooth acquisition optimisation (correct).

- **Cost:** 5× current TabPFN inference (linear in candidates) ≈ +5 min per run
- **Acquisition:** standard `optimize_acqf`, gradient fully correct
- **Implementation effort:** Low

### Method 2: TFM-FSBO (TabPFN Embedding as GP Feature Space)

Distil TabPFN's penultimate embedding `φ_TFM(x)` into a small MLP `φ_MLP(x)`.
Fit GP in the MLP feature space: `k(x,x') = k_GP(φ_MLP(x), φ_MLP(x'))`.
Optimise acquisition via gradient through `φ_MLP + GP` — fully differentiable, no approximation.

- Inspired by DKL (Wilson 2016) and FSBO (Wistuba 2021)
- **Acquisition:** gradient through MLP + GP, correct by construction
- **Implementation effort:** High

### Method 3: TFM-MTBO-Covar (TabPFN-Derived Task Covariance) ← strongest contribution

Replace the `IndexKernel`'s MLL-estimated inter-task correlation with a **TabPFN cross-predictive
similarity** computed per BO step:

```
NLL(t1→t2) = TabPFN NLL of y_t2 | context=(X_t1, y_t1), query=X_t2
NLL(t2→t1) = TabPFN NLL of y_t1 | context=(X_t2, y_t2), query=X_t1

ρ_12 = symmetrised similarity from both NLLs   → fixed into Σ_task
```

MLL only fits lengthscales + per-task variance (not ρ). Everything else is standard MultiTaskGP.
Acquisition optimisation is **completely unchanged** — the GP is a valid kernel machine.

**Key properties:**
- Self-calibrating transfer: ρ → 1 for CI problems, ρ → 0 for NI problems automatically
- Negative transfer protection: dissimilar tasks → ρ ≈ 0 → independent GPs (graceful fallback)
- Only 2 extra TabPFN forward passes per step (T=2 tasks)
- Interpretable: plot ρ trajectory over steps as a diagnostic

**Variant:** asymmetric directional weighting replaces the fixed elite ratio with a per-step
TFM-derived transfer weight `w(t_source → t_target) = s(t_source → t_target)`.

## 5. Revised Paper Framing

The paper cannot claim "TabPFN is a better surrogate" — the 10D and 50D results do not support this.
The defensible, publishable claim is:

> **TabPFN's pre-trained meta-knowledge is best exploited as a prior over task relationships
> (Method 3) or as a candidate ranker (Method 1), not as a direct surrogate replacement.
> Used as a drop-in surrogate, TabPFN's non-smooth ensemble predictions degrade acquisition
> function optimisation, particularly in sparse high-dimensional regimes.**

Supporting narrative structure:
1. Empirical result: current TFM-as-surrogate fails at 50D (acquisition landscape analysis)
2. Diagnosis: ensemble non-smoothness + 50D sparsity → wiggly LCB → optimizer failure
3. Method 3: data-driven ρ from TabPFN cross-predictions fixes IndexKernel's core weakness
4. Method 1: screening exploits TabPFN's ranking ability without requiring smoothness
5. Ablation: ρ trajectory on CI vs NI problems — validates the mechanism
